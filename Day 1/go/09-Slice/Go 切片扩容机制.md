



## Go 切片扩容机制

这是一个非常好的问题！这涉及到 Go 语言的**扩容策略**。

### 扩容规则

Go 的切片扩容策略（源码位于 `runtime/slice.go`）：

| 当前容量 | 扩容方式 |
|---------|---------|
| **< 1024** | 翻倍扩容 (`cap * 2`) |
| **≥ 1024** | 缓慢增长 (`cap * 1.25`) |

### 你的例子分析

```
追加前: len=2, cap=2, 切片=[0,1]
追加3个元素: 需要总容量 >= 5 (原有2 + 新增3)
```

按照**翻倍策略**：
- 2 × 2 = 4 ❌ 4 < 5，不够
- 需要找到**第一个 ≥ 5 的数** → **6** ✓

所以容量变成 6。

### 为什么不直接扩容到 5？

Go 的扩容不是精确计算，而是**按倍数预分配**，目的是：

1. **减少频繁扩容** - 预留空间，避免下次追加又扩容
2. **内存对齐** - Go 会根据元素类型做内存对齐
3. **性能优化** - 减少内存分配次数

### 验证扩容规律

你可以用代码验证：

```go
package main

import "fmt"

func main() {
    var s []int
    
    // 不断追加，观察容量变化
    for i := 0; i < 20; i++ {
        s = append(s, i)
        fmt.Printf("len=%d, cap=%d\n", len(s), cap(s))
    }
}
```

**输出：**
```
len=1, cap=1
len=2, cap=2
len=3, cap=4    ← 2*2=4 > 3
len=4, cap=4
len=5, cap=8    ← 4*2=8 > 5
...
len=17, cap=32  ← 16*2=32 > 17
```

### 总结

| 问题 | 答案 |
|-----|------|
| 为什么是 6 不是 5？ | 翻倍策略下，2×2=4 不够，需要 6 |
| 为什么不多不少？ | 找到第一个**≥需求容量**的数 |
| 为什么要翻倍？ | 减少扩容次数，提升性能 |
