
## 问题分析

这是一个关于 **Go 协程执行顺序** 的问题。让我解释为什么把 `fibonacci(c, quit)` 放到 `go func()` 上面会报错（或死锁）。

### 当前代码的执行流程

```go
func main() {
    c := make(chan int)
    quit := make(chan int)
    
    fibonacci(c, quit)  // ❌ 这是阻塞调用！

    go func() {        // 这行永远不会执行到
        for i := 0; i < 5; i++ {
            fmt.Println(<-c)
        }
        quit <- 0
    }()
}
```

### 为什么会死锁？

关键在于 `fibonacci` 函数的实现：

```go
func fibonacci(c, quit chan int) {
    x, y := 0, 1
    for {
        select {
        case c <- x:    // 发送数据到 c 通道
            x, y = y, x+y
        case <-quit:    // 接收退出信号
            return
        }
    }
}
```

当你调用 `fibonacci(c, quit)` 时：

1. **主 goroutine 进入 `fibonacci` 函数**，开始执行无限循环
2. `select` 语句等待两个事情：
    - 有人接收 `c` 通道的数据 (`<-c` 在接收端)
    - 有人发送数据到 `quit` 通道
3. **此时没有其他 goroutine 在运行**，所以：
    - `c` 通道没有人接收 → 发送阻塞
    - `quit` 通道没有发送数据 → 接收阻塞
4. **死锁！** 程序卡住无法继续

---

### 正确做法

把 `fibonacci(c, quit)` 放到 `go func()` 中启动为协程：

```go
func main() {
    c := make(chan int)
    quit := make(chan int)

    // ✅ 启动一个 goroutine 来运行 fibonacci
    go fibonacci(c, quit)

    // 主 goroutine 接收数据
    go func() {
        for i := 0; i < 5; i++ {
            fmt.Println(<-c)
        }
        quit <- 0
    }()
    
    // 需要添加等待逻辑，否则主程序会直接退出
}
```

### 总结

| 调用方式 | 后果 |
|---------|------|
| `fibonacci(c, quit)` | 主 goroutine 阻塞在 select，永远等待数据被接收 |
| `go fibonacci(c, quit)` | fibonacci 在后台运行，不阻塞主 goroutine |

**核心原因**：Go 的 channel 通信是同步的，发送端会阻塞直到有接收端，反之亦然。`fibonacci` 需要作为 goroutine 运行，同时有另一个 goroutine 接收数据，这样才能形成正常的通信循环。
