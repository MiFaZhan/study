
这是个很好的问题！确实，如果把所有商品都缓存到 Redis，内存消耗会很大。但实际生产中有几个关键策略来控制 Redis 大小：

## 1. **只缓存热数据，不是全量缓存**

```
┌─────────────────────────────────────────┐
│  商品总数：100万件                        │
│  Redis只缓存：1万件（前1%的热门商品）      │
│  内存占用：~500MB（可控）                 │
└─────────────────────────────────────────┘
```

**核心思想**：
- 80% 的流量集中在 20% 的商品上（二八定律）
- 只缓存这 20% 的热商品就够用了
- 冷商品直接查数据库，反正访问量低

## 2. **设置过期时间，自动淘汰**

```go
// 商品缓存 10 分钟过期
rdb.Set(ctx, cacheKey, data, 10*time.Minute)

// 热门商品会被反复访问，过期后会自动重新加载
// 冷门商品过期后不再加载，自然淘汰
```

**效果**：
- 热商品：一直在 Redis 中（反复刷新）
- 冷商品：过期后不再占用内存

## 3. **Redis 的内存淘汰策略**

当 Redis 内存达到上限时，会自动删除旧数据：

```bash
# Redis 配置
maxmemory 2gb                    # 限制最大内存
maxmemory-policy allkeys-lru      # LRU 算法淘汰最少使用的 key
```

**常见策略**：
- `allkeys-lru`：删除最近最少使用的 key（推荐）
- `volatile-lru`：只删除设置了过期时间的 key
- `allkeys-lfu`：删除访问频率最低的 key

## 4. **实际内存占用估算**

假设缓存 10,000 个商品：

```go
type Product struct {
    ID    int     `json:"id"`        // 4 bytes
    Name  string  `json:"name"`      // ~50 bytes
    Price float64 `json:"price"`     // 8 bytes
    Stock int     `json:"stock"`     // 4 bytes
}
// JSON 序列化后约 100 bytes/个
// 10,000 个商品 ≈ 1 MB（实际加上 Redis 开销约 5-10 MB）
```

**结论**：即使缓存 10 万个热商品，也就 50-100 MB，完全可控。

## 5. **监控和动态调整**

```go
// 定期检查 Redis 内存使用情况
func MonitorRedisMemory() {
    info := rdb.Info(ctx, "memory").Val()
    // 解析 used_memory_human 字段
    // 如果超过阈值，可以：
    // 1. 减少缓存时间
    // 2. 只缓存更热的数据
    // 3. 增加内存限制
}
```

## 总结

| 策略 | 说明 |
|------|------|
| **只缓存热数据** | 20% 的商品承载 80% 流量 |
| **设置过期时间** | 冷商品自动淘汰 |
| **LRU 淘汰** | 内存满时自动删旧数据 |
| **监控调整** | 根据实际情况动态优化 |

**Redis 不会无限变大**，它是一个有容量限制、自动淘汰的缓存层，不是永久存储数据库。
