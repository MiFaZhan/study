
## GoFrame 项目架构概览

这是一个基于 GoFrame v2.7.1 的标准单体应用（SingleRepo）项目，采用了经典的分层架构设计。

本项目是一个单词管理系统，包含用户注册/登录、JWT 鉴权、单词 CRUD 等完整功能。

### 分层架构详解

#### 1. API 层 (api/)
**职责：** 定义对外的 API 接口契约

**特点：**
- 使用 `g.Meta` 标签定义路由、HTTP 方法、文档说明
- 使用 `v` 标签定义验证规则（required、length、email、between 等）
- 使用 `dc` 标签定义字段描述（用于生成 API 文档）
- 支持版本管理（v1, v2, v3...）
- 通过 `gf gen ctrl` 命令自动生成接口定义和 Controller 骨架

---

#### 2. Controller 层 (internal/controller/)
**职责：** 接收 HTTP 请求，调用业务逻辑，返回响应

**文件组织模式：**
- `{module}.go`：手动编写的业务代码（可自由修改）
- `{module}_new.go`：自动生成的构造函数（不可修改）
- `{module}_v1_{method}.go`：自动生成的接口实现骨架（可修改）

**特点：**
- 实现 API 层定义的接口
- 通过 `cmd.go` 中的 `group.Bind()` 自动注册路由
- 使用 `MiddlewareHandlerResponse` 中间件统一处理响应格式
- Controller 层通常只做参数传递，具体业务逻辑在 Logic 层实现

---

#### 3. Logic 层 (internal/logic/)
**职责：** 实现复杂的业务逻辑

**特点：**
- 实现 Service 层定义的接口（如果有）
- 可被多个 Controller 复用
- 不直接处理 HTTP 请求，专注于业务逻辑
- 调用 DAO 层进行数据库操作
- 包含业务规则、事务处理、数据组装等

---

#### 4. Service 层 (internal/service/)
**职责：** 定义业务服务的接口（interface）

**当前状态：** 本项目暂未使用 Service 层，Controller 直接调用 Logic

**为什么 Service 目录是空的？**

本项目采用了**简化开发模式**，Controller 直接依赖 Logic 具体实现：

```go
// 当前做法：Controller 直接依赖 Logic 实现
import (
    usersLogic "star/internal/logic/users"
    wordsLogic "star/internal/logic/words"
)

type ControllerV1 struct {
    users *usersLogic.Users  // 直接依赖具体类型
    words *wordsLogic.Words
}

func (c *ControllerV1) Create(ctx context.Context, req *v1.CreateReq) {
    err := c.words.Create(ctx, ...)  // 直接调用 Logic 方法
    return nil, err
}
```

**GoFrame 标准开发模式（推荐）：**

GoFrame 官方推荐使用 Service 接口层来解耦 Controller 和 Logic：

1. **先在 Logic 层实现业务逻辑**
   ```go
   // internal/logic/words/words.go
   type Words struct {}
   
   func (w *Words) Create(ctx context.Context, in CreateInput) error {
       // 业务逻辑实现
   }
   ```

2. **运行命令自动生成 Service 接口**
   ```bash
   gf gen service  # 或 make service
   ```
   
   这会扫描 Logic 层的公开方法，自动生成接口定义：
   ```go
   // internal/service/words.go（自动生成）
   type IWords interface {
       Create(ctx context.Context, in logic.CreateInput) error
       Update(ctx context.Context, in logic.UpdateInput) error
       // ... 其他方法
   }
   
   func Words() IWords {
       return logic.New()  // 返回 Logic 实现
   }
   ```

3. **Controller 通过 Service 接口调用**
   ```go
   // internal/controller/words/words_v1_create.go
   import "star/internal/service"
   
   func (c *ControllerV1) Create(ctx context.Context, req *v1.CreateReq) {
       // 通过接口调用，不依赖具体实现
       err := service.Words().Create(ctx, ...)
       return nil, err
   }
   ```

**两种模式对比：**

| 特性 | 简化模式（当前） | 标准模式（推荐） |
|------|-----------------|-----------------|
| Controller 依赖 | Logic 具体实现 | Service 接口 |
| 代码耦合度 | 较高 | 低 |
| 单元测试 | 较难 Mock | 易于 Mock |
| 依赖注入 | 不支持 | 支持 |
| 开发速度 | 快（少一层） | 稍慢 |
| 适用场景 | 小型项目、快速原型 | 中大型项目、团队协作 |

**设计原则：**
- 只包含方法签名，不包含具体实现
- 遵循"面向接口编程"原则
- Controller 依赖 Service 接口，而非直接依赖 Logic 实现
- 便于单元测试和依赖注入
- 支持多种 Logic 实现切换（如测试环境使用 Mock 实现）

**生成方式：**
```bash
make service  # 或 gf gen service
```

**何时需要 Service 层？**
- ✅ 项目规模较大，需要更好的分层
- ✅ 需要编写单元测试，需要 Mock Logic 层
- ✅ 多人协作开发，需要明确接口契约
- ✅ 需要支持依赖注入框架
- ❌ 快速原型开发，追求开发速度
- ❌ 小型项目，业务逻辑简单

---

#### 5. DAO 层 (internal/dao/)
**职责：** 数据访问对象，封装数据库操作

**实际生成的 DAO：**
- `users.go`：Users 表的 DAO 对外接口
- `words.go`：Words 表的 DAO 对外接口
- `internal/users.go`：Users 表的 DAO 内部实现（自动生成）
- `internal/words.go`：Words 表的 DAO 内部实现（自动生成）

**生成方式：**
```bash
make dao  # 或 gf gen dao
```

**特点：**
- 根据数据库表结构自动生成
- 提供类型安全的 CRUD 接口
- 配合 Model 层的 Entity 和 DO 使用
- 支持链式调用、条件构建、分页查询等

---

#### 6. Model 层 (internal/model/)
**职责：** 数据模型定义

**实际生成的模型：**

**entity/（数据库表实体）：**
- `users.go`：Users 表实体
  ```go
  type Users struct {
      Id        uint        `json:"id"`
      Username  string      `json:"username"`
      Password  string      `json:"password"`
      Email     string      `json:"email"`
      CreatedAt *gtime.Time `json:"createdAt"`
      UpdatedAt *gtime.Time `json:"updatedAt"`
  }
  ```
- `words.go`：Words 表实体
  ```go
  type Words struct {
      Id                 uint        `json:"id"`
      Uid                uint        `json:"uid"`  // 用户 ID
      Word               string      `json:"word"`
      Definition         string      `json:"definition"`
      ExampleSentence    string      `json:"exampleSentence"`
      ChineseTranslation string      `json:"chineseTranslation"`
      Pronunciation      string      `json:"pronunciation"`
      ProficiencyLevel   uint        `json:"proficiencyLevel"`
      CreatedAt          *gtime.Time `json:"createdAt"`
      UpdatedAt          *gtime.Time `json:"updatedAt"`
  }
  ```

**do/（数据操作对象）：**
- `users.go`：用于 Users 表的查询条件和更新数据
- `words.go`：用于 Words 表的查询条件和更新数据

**生成方式：**
```bash
make dao  # 同时生成 DAO、Entity、DO
```

#### 7. 其他重要目录

**packed/ - 资源打包**

- `hack.mk`：Make 命令定义（构建、生成代码、部署）
- `hack-cli.mk`：CLI 工具自动安装脚本

**manifest/i18n/ - 国际化**
- `zh-CN/validation.toml`：验证错误信息中文化
  ```toml
  "gf.gvalid.rule.required" = "{field}字段不能为空"
  "gf.gvalid.rule.email"    = "{field}字段值`{value}`邮箱地址格式不正确"
  "gf.gvalid.rule.length"   = "{field}字段值`{value}`字段长度应当为{min}到{max}个字符"
  ```

### 关键特性

1. **自动路由绑定**
   - 通过 `g.Meta` 标签定义路由信息
   - 使用 `group.Bind()` 自动注册路由
   - 支持 RESTful 风格和自定义路径

2. **代码自动生成**
   - `make ctrl`：根据 API 定义生成 Controller 骨架
   - `make dao`：根据数据库表生成 DAO/Entity/DO
   - `make service`：根据 Logic 实现生成 Service 接口（本项目未使用）
   - `make enums`：解析代码生成枚举类型

3. **标准化响应**
   - 使用 `MiddlewareHandlerResponse` 中间件统一处理响应
   - 自动封装成功/失败响应格式
   - 支持自定义响应结构

4. **配置管理**
   - YAML 配置文件（`manifest/config/config.yaml`）
   - 支持多环境配置（开发/测试/生产）
   - 配置热更新（部分配置项）

5. **OpenAPI 支持**
   - 自动生成 API 文档（基于 `g.Meta` 标签）
   - Swagger UI 访问路径：`http://localhost:8000/swagger`
   - API JSON 访问路径：`http://localhost:8000/api.json`

6. **资源打包**
   - 通过 `internal/packed` 将配置和资源文件打包到二进制
   - 支持单文件部署，无需额外配置文件

7. **构建和部署**
   - Makefile 提供丰富的命令（构建/测试/部署）
   - Docker 支持（自动构建镜像）
   - Kubernetes 部署配置（Kustomize）

### 开发流程示例

**添加新的 API 接口：**

1. 定义 API 接口（`api/user/v1/user.go`）
   ```go
   type CreateReq struct {
       g.Meta `path:"/user" method:"post" summary:"创建用户" tags:"用户"`
       Name   string `v:"required|length:3,12" json:"name" dc:"用户名"`
       Email  string `v:"required|email" json:"email" dc:"邮箱"`
   }
   type CreateRes struct {
       Id int64 `json:"id" dc:"用户ID"`
   }
   ```

2. 生成 Controller 骨架
   ```bash
   make ctrl
   # 自动生成：
   # - api/user/user.go（接口定义）
   # - internal/controller/user/user_new.go（构造函数）
   # - internal/controller/user/user_v1_create.go（实现骨架）
   ```

3. 实现业务逻辑（`internal/logic/user/user_create.go`）
   ```go
   func (u *User) Create(ctx context.Context, req *v1.CreateReq) (res *v1.CreateRes, err error) {
       // 实现创建用户逻辑
       result, err := dao.Users.Ctx(ctx).Data(do.Users{
           Name:  req.Name,
           Email: req.Email,
       }).Insert()
       
       if err != nil {
           return nil, err
       }
       
       id, _ := result.LastInsertId()
       return &v1.CreateRes{Id: id}, nil
   }
   ```

4. Controller 调用 Logic（`internal/controller/user/user_v1_create.go`）
